**作成日**: 2026-03-01
**バージョン**: 0.2（レビュー反映版）
**変更履歴**: v0.1 → v0.2: UX/技術/ドメイン3観点の専門家レビューを反映

## 1. プロダクトビジョン

> **「法律の専門知識がなくても、マンション理事会の誰もが自信を持って管理規約の改正を進められる」**

2026年4月の改正区分所有法施行に伴い、全国約12万の管理組合が規約改正を迫られている。専門家に委託すると46〜58万円、管理会社に依頼しても10万円弱かかる一方、法律の素人である理事会メンバーが自力で進めるには知識・時間の壁が大きい。

本プロダクトは、AI（Claude API）を活用して、この「専門家と自力の間」を埋めるセルフサービス型の規約改正支援ツールである。

**重要な位置付け**: 本サービスは法律相談・法律事務を行うものではなく、一般的な情報提供およびドラフト補助を目的とする。生成された条文案は専門家の確認を経た上で使用することを推奨する。

## 2. 市場環境

### 2.1 タイミングの切迫性

| 日付 | イベント |
|------|----------|
| 2024年5月 | 改正区分所有法 成立・公布 |
| 2024年6月 | 国交省 標準管理規約を改正（令和6年改正: 置き配、EV充電等） |
| 2025年10月17日 | 国交省 標準管理規約を全面改正（令和7年改正: 改正法対応） |
| **2026年4月1日** | **改正区分所有法 施行** |
| 2026年4〜6月 | 多くのマンションの通常総会シーズン |

**施行後（パターン2）の規約改正が現実的なメインシナリオ。**
施行後は新法の緩和された決議要件（出席者3/4）が使えるため、むしろ施行後に改正するメリットがある。

**法的な正確な整理**: 施行後は、規約の決議要件に関する条文がそのまま残っていても、改正法の規定が優先して適用される（強行規定の優先）。規約を放置しても管理組合の運営は可能だが、規約と法律の内容が食い違う状態が続くため、早めの規約改正を推奨する。

**施行前対応（パターン1）の場合**: 旧法の決議要件（組合員総数・議決権総数の各3/4以上）が適用される。招集手続を施行日前に開始した場合は旧法が適用される点に注意。

### 2.2 法改正の3層構造

本プロダクトが扱う法改正は以下の3層で整理される。

| 層 | 内容 | 時期 | 性格 |
|----|------|------|------|
| 第1層 | **改正区分所有法** | 2024年成立、2026年4月施行 | 法律レベルの変更 |
| 第2層 | **令和6年改正 標準管理規約** | 2024年6月 | 生活利便性向上（置き配、EV充電等） |
| 第3層 | **令和7年改正 標準管理規約** | 2025年10月 | 改正法への全面対応（第2層を包含・上書き） |

ギャップ分析（F3）の基準データは**第3層（令和7年改正）**を使用する。第2層は第3層に包含されているため個別管理は不要。

## 3. ユーザージャーニーと機能定義

### 3.1 ジャーニーマップ

```
[0.始める]  → [1.理解する] → [2.現状を把握する] → [3.差分を分析する] → [4.改正案を作る] → [5.合意を形成する]
 初回登録・     法改正の      現行規約を          AIが標準規約と     AIがドラフトを    説明資料・
 属性ヒアリング  概要を学ぶ    アップロード         比較・ギャップ特定   生成・レビュー    議案を生成
 (5分)         (15分)       (10分)             (自動5分+確認10分)  (60〜90分)       (10分)
```

各ステップの所要時間目安をUI上に常時表示し、ユーザーの不安を軽減する。

### 3.2 全画面共通UI

| 要素 | 仕様 |
|------|------|
| **ジャーニー進捗バー** | 全画面のヘッダーに6ステップの進捗バーを常時表示。現在のステップをハイライト |
| **文字サイズ調整** | レビューモード画面上に常時表示の「文字サイズ A⁻ A⁺」ボタン。CSSの`rem`ベースで制御 |
| **AIチャット（F5）** | サイドパネルまたはフローティングウィジェットとして常時アクセス可能。現在表示中の条文がコンテキストとして自動で渡される |
| **自動保存** | 全操作はFirestoreにリアルタイム同期。明示的な保存操作は不要 |
| **再訪時** | 「前回の続きから再開しますか？」のダイアログ表示。前回のステップ・条文位置に直接復帰可能 |

#### カード内の4層構造

各条文カードは以下の4層で構成し、上から順に読めば判断できるようにする。

| 層 | 内容 | 表示 |
|----|------|------|
| **① 平易な要約** | 「何が変わるのか」「なぜ変えるのか」を2〜3行の日常語で説明。AIが生成 | 常に表示 |
| **② 新旧対照** | 現行条文と改正案を並べ、変更箇所をハイライト（追加=青背景＋「＋」記号、削除=赤取消線、修正=黄背景＋「→」記号）。PC: 左右2列、スマホ: 上下配置に自動切替 | 折りたたみ可（デフォルト展開） |
| **③ 理事会向け説明文** | 住民や他の理事への説明に使えるフレーズをAIが提案 | 折りたたみ（デフォルト非表示） |
| **④ 判断ボタン + メモ** | 採用 / 修正 / 保留の3択 + 自由記入メモ欄 + 「この条文について質問する」リンク（F5チャットを起動） | 常に表示 |

#### 「修正」選択時のフロー

「修正」ボタンを押すと、以下の分岐UIを表示する:

1. **よくある修正パターンの選択肢**をAIが提案（例: 「数値を変更したい」「対象範囲を限定したい」「表現をもう少し厳しく/緩くしたい」）
2. **自然言語で修正意図を入力** → AIが条文を再生成 → プレビュー表示
3. **条文テキストの直接編集**（上級者向け、トグルで表示）

修正後は「やり直し」（前の状態に戻す）が可能。修正履歴を保持する。

#### 「保留」選択時のセーフガード

- **重要度「法的必須」の項目**に「保留」が選択された場合、確認ダイアログを表示: 「この項目は法改正への対応として必須です。保留にした場合、以下のリスクがあります: [具体的リスク]。それでも保留にしますか？」
- 全レビュー完了時に、保留中の「法的必須」項目が残っている場合はサマリー画面で警告
- エクスポート時、保留の「法的必須」項目を別枠で一覧化し注意喚起

#### AI生成条文の品質担保

| 方式 | 説明 |
|------|------|
| **テンプレートベース生成** | 標準管理規約の条文をベース（テンプレート）とし、マンション固有情報を差し込む方式を基本。AIの自由度が高い「完全新規起草」は最小限に抑え、その場合は警告レベルを引き上げ |
| **強行規定チェック** | 生成された条文が区分所有法の強行規定に抵触していないかをルールベースで自動検証。抵触時は警告表示 |
| **出典明示** | 各ドラフト条文に「標準管理規約 第○条をベースに生成」と出典を表示。トレーサビリティを確保 |
| **免責表示** | 全条文に「AI生成のドラフトです。採用前に専門家の確認を推奨します」を付記 |

#### アクセシビリティ・視認性

| 項目 | 仕様 |
|------|------|
| フォントサイズ | 本文16px以上、条文テキスト18px以上。画面上の「A⁻ A⁺」ボタンで拡大可（最大24px） |
| コントラスト | WCAG 2.1 AA準拠（コントラスト比4.5:1以上） |
| 色だけに頼らない | 変更箇所は色＋記号（追加: 「＋」、削除: 取消線、修正: 「→」）で表現 |
| 印刷最適化 | 印刷時は白黒でも判読できるレイアウトに自動変換。A4縦・余白広め・フォント12pt以上 |
| 進捗の見える化 | 上部に「3/28 完了」のプログレスバー。未判断の条文にはマークを表示 |
| タッチターゲット | ボタン・タップ領域は最小44x44px（WCAG 2.1準拠） |
| モバイル対応 | 新旧対照を上下配置に切替。「前へ/次へ」ボタンを常時大きく表示 |
| セッション管理 | 「ここまで保存して終了」ボタン。再訪時は前回の条文位置から自動再開 |

### 3.5 エクスポート形式（F6）

## 5. 技術アーキテクチャ（GCPベース）

### 5.1 概要

### 5.2 技術選定の理由

| 選択 | 理由 |
|------|------|
| **Next.js on Cloud Run** | GCPネイティブ。SSR/API Routes統合。Cloud RunならAPI Routeのタイムアウト上限に縛られず長時間処理（30秒〜数分）も実行可能 |
| **Firestore** | スキーマレスで規約の構造化データに柔軟対応。ベクトル検索機能あり（将来のRAG拡張に備え）。GCPネイティブ。無料枠あり |
| **Claude API (Sonnet 4.5)** | 日本語の法律文書に対する理解力が高い。200Kコンテキストウィンドウ。$3/$15 per MTokでコスト対性能比が良い |
| **Firebase Auth** | GCPネイティブ。Google認証で手軽にログイン。メール認証も対応 |
| **Google Document AI** | GCPネイティブ。表構造・番号付きネストの認識精度が高い。日本語OCR対応。無料枠: 月1,000ページ |
| **puppeteer (PDF生成)** | HTML/CSSで新旧対照表レイアウトをそのまま組める。印刷用CSS制御も容易。@sparticuz/chromiumで軽量化 |

### 5.3 Claude API の利用設計

| 機能 | モデル | プロンプト戦略 |
|------|--------|----------------|
| 規約パース | Haiku 4.5 | Document AI出力の後処理として条文構造を認識・構造化JSON化。コスト重視 |
| ギャップ分析 | Sonnet 4.5 | **章単位で分割分析**（第1章〜第8章）。各チャンクで「標準規約の該当章＋ユーザー規約の該当章」のペアを入力。1回あたり15,000〜25,000トークンに抑制 |
| ドラフト生成 | Sonnet 4.5 | **テンプレートベース**。ギャップ分析結果（構造化済み）＋標準規約の該当条文＋現行条文を入力し、改正案を生成。原文全文の再送を避ける |
| チャットQ&A | Sonnet 4.5 | **MVPではコンテキスト全詰め込み方式**。基準データ（約40,000トークン）を毎回システムプロンプトに含め回答。RAGインフラ不要で実装が早く、法律文書の相互参照が多い特性に適合。将来はFirestoreベクトル検索によるRAGへ移行可能 |
| ストリーミング | 全モデル | SSE（Server-Sent Events）でリアルタイム表示。処理中は「現在〇〇を処理中です」とステータスメッセージ表示 |

## 7. リスクと対策

### 7.1 リスク一覧

| リスク | 影響度 | 発生確率 | 対策 |
|--------|--------|----------|------|
| **AIが法律の強行規定に反する条文を生成** | 高 | 中 | テンプレートベース生成＋強行規定チェッカーで自動検証。出典条文番号を必ず出力させ実在を機械検証 |
| **非弁行為（弁護士法72条）に抵触** | 高 | 低 | §7.2で詳述 |
| **AIがハルシネーション（存在しない条文を参照等）** | 高 | 中 | 出典条文番号をバリデーション。基準データのみに基づく回答。全条文に「AI生成ドラフト」と免責表示 |
| **PDFパースで条文構造が正しく認識されない** | 中 | 高 | Document AIをプライマリに、pdf-parse + Haikuをフォールバック。パース結果の確認ステップUIを必須化 |
| **Claude APIの応答時間が長くUX悪化（30秒超）** | 中 | 中〜高 | SSEストリーミングでリアルタイム表示。分析をチャンク分割して進捗表示 |
| **施行日に間に合わない** | 中 | 低 | パターン2（施行後対応）が主流のため、4月以降のローンチでも価値あり |
| **Claude APIのコスト超過** | 中 | 低 | Haiku活用のコスト最適化。分析結果のキャッシュ（Firestore永続化）。1ユーザーあたり上限設定 |

### 7.2 弁護士法72条（非弁行為）対応方針

**法的整理**: AIによる条文ドラフト生成は、以下の理由で弁護士法72条の「法律事務の取扱い」には原則該当しない:
- 規約改正は「法律事件」（紛争性のある案件）ではなく、管理組合の自治的な規範定立行為
- AIは「情報提供・テンプレート補助」であり、法律判断を行わない
- 最終判断はあくまで理事会・総会が行う

**ただし非弁リスクが高まるケース**:
- AIが個別の紛争に関して法的助言を行う場合
- AIが「訴訟すべき」等の法的判断を提示する場合

**実装上の対策**:
1. **利用規約・免責条項**: 「本サービスは法律相談・法律事務を行うものではなく、一般的な情報提供およびドラフト補助を目的とする」を明記
2. **F5チャットのガードレール**: 個別紛争に関する質問（「滞納者への法的措置は？」「隣人トラブルの解決方法は？」等）を検知し、回答を制限して専門家への相談を促す
3. **全画面フッター**: 「AIの出力はあくまで参考情報であり法的助言ではありません」を常時表示
4. **ドラフト条文の免責表示**: 全条文に「標準管理規約を参考にしたAI生成ドラフトです。採用前に専門家の確認を推奨します」を付記
